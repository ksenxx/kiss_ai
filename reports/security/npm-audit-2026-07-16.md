# npm vulnerability investigation

**Snapshot date:** 2026-07-16 UTC\
**Scope:** npm packages locked by KISS Sorcar 2026.7.23\
**Tooling:** Node.js 25.8.1, npm 11.11.0

## Executive summary

A clean, lockfile-based audit of KISS Sorcar's only tracked npm project reports **4 vulnerable package objects: 3 Low and 1 High**. This is not four independent security flaws. It is one legacy dependency chain ending in one vulnerable installed package with two advisories:

```text
gts@7.0.0 (direct development dependency)
└─ inquirer@7.3.3
   └─ external-editor@3.1.0
      └─ tmp@0.0.33  ← vulnerable installed copy
```

npm propagates the `tmp` vulnerability through `external-editor`, `inquirer`, and `gts` as “meta-vulnerabilities,” producing four package objects. The two underlying advisories both affect `tmp@0.0.33`:

| Advisory | Severity used by npm | Issue | Affected | Fixed |
|---|---:|---|---:|---:|
| [GHSA-ph9p-34f9-6g65](https://github.com/advisories/GHSA-ph9p-34f9-6g65) / CVE-2026-44705 | High | Path traversal via untrusted `prefix`, `postfix`, or `dir` | `<0.2.6` | `0.2.6` |
| [GHSA-52f5-9888-hmc6](https://github.com/advisories/GHSA-52f5-9888-hmc6) / CVE-2025-54798 | Low | Symlink-based temporary file/directory write via `dir` | `<=0.2.3` | `0.2.4` |

**User/runtime exposure is low:** every declared npm package is a `devDependency`; `npm audit --omit=dev` reports **0 vulnerabilities**; the VSIX is built with `--no-dependencies`; and KISS application code does not import or call this chain. The vulnerable copy is installed on developer and release-build machines, however, so the result should not be dismissed.

**Recommended remediation:** force only `external-editor`'s `tmp` to the current safe `0.2.7` with an npm override, then run the complete extension checks. An isolated reproduction of that override produced **0 audit findings**, loaded `gts@7.0.0`, and passed an `external-editor` temporary-file smoke test. This report does not change the lockfile or silently adopt that override.

## Scope and reproduction

The repository has one tracked npm manifest and lockfile:

```text
src/kiss/agents/vscode/package.json
src/kiss/agents/vscode/package-lock.json
```

All declared packages are under `devDependencies`; there is no `dependencies` section. The following checks were performed against the committed lockfile:

```bash
cd src/kiss/agents/vscode
npm ci --ignore-scripts --omit=optional
npm audit --json
npm audit --omit=dev --json
npm explain tmp
npm ls gts inquirer external-editor tmp
npm audit fix --dry-run --json
```

Results:

| Check | Result |
|---|---|
| Full `npm audit` | Exit 1; 4 package objects: 3 Low, 1 High |
| `npm audit --omit=dev` | Exit 0; 0 vulnerabilities |
| Clean `npm ci` post-install audit | Reproduced 4 findings |
| `npm audit fix --dry-run` | No remediation; `fixAvailable: false` throughout the chain |

Audit data changes as registry advisories change, so these numbers are a dated snapshot rather than a permanent property of the lockfile.

## What is actually vulnerable

The vulnerable node is exactly:

```text
node_modules/external-editor/node_modules/tmp@0.0.33
```

The lockfile also contains a separate `tmp@0.2.7` used by `@vscode/vsce@3.9.2`. That copy is outside both affected ranges and is not the finding.

`npm audit` reports these objects:

| Package object | Direct? | npm severity | Why it appears |
|---|---:|---:|---|
| `tmp` | No | High | The installed `0.0.33` is directly affected by both advisories. |
| `external-editor` | No | Low | Its dependency range resolves only to the vulnerable legacy `tmp` line. |
| `inquirer` | No | Low | It depends on the affected `external-editor`. |
| `gts` | Yes | Low | It depends on the affected `inquirer`; this is the only direct package in the chain. |

The package-object severities above are npm's output. Advisory databases have some scoring differences: GitHub assigns the symlink advisory 2.5 Low while NVD currently shows both that CNA score and its own 5.3 Medium score. For CVE-2026-44705, GitHub uses CVSS v4 7.7 High while NVD currently uses CVSS v3.1 8.2 High. These differences do not change the affected or fixed versions.

Some advisory prose still says that no patched version is available. That prose is stale: GitHub's structured package tables, OSV's machine-readable ranges, NVD, and the npm registry consistently identify `0.2.6` as fixing the High advisory; current `tmp` is `0.2.7`.

## Exposure assessment

### Installed extension and end users

No vulnerable production dependency was found:

- `package.json` declares only development dependencies.
- `npm audit --omit=dev` returns zero findings.
- VSIX packaging invokes `scripts/package-vsix.js --no-dependencies`, so development `node_modules` are not shipped as extension dependencies.
- Searches of tracked application code found no import or invocation of `gts`, `inquirer`, `external-editor`, or `tmp`.
- KISS invokes ESLint directly for routine linting. `gts` supplies development style/configuration, including the Prettier configuration.

Accordingly, there is no identified path for an extension user or model prompt to pass attacker-controlled `prefix`, `postfix`, or `dir` values into the vulnerable `tmp` copy.

### Developer and release machines

The chain is physically installed during source setup and releases:

- `install.sh` runs `npm ci --ignore-scripts --omit=optional ...`.
- `scripts/release.sh` and `scripts/release_exp.sh` run equivalent development installs.
- `--ignore-scripts` reduces package-install attack surface, but it does not remove this package or repair its runtime path handling.
- Those flows pass `--no-audit`, suppressing npm's post-install vulnerability summary. `scripts/release.sh` has an older comment describing low-severity `gts`/`inquirer` findings, but the new 2026 High advisory makes that comment incomplete.

Exploitation of the High advisory requires a caller to pass untrusted data into affected temporary-path options. The Low advisory additionally requires a suitable attacker-controlled symlink. No such KISS call site was found, making practical risk low, but build-host exposure remains non-zero because the vulnerable code is installed and could be reached through interactive tooling or future code changes.

## Remediation analysis

### 1. Targeted npm override (recommended short-term)

The smallest candidate change is:

```json
{
  "overrides": {
    "encoding-sniffer": "^1.0.2",
    "external-editor": {
      "tmp": "0.2.7"
    }
  }
}
```

This was tested in an isolated package containing `gts@7.0.0`:

```text
gts@7.0.0
└─ inquirer@7.3.3
   └─ external-editor@3.1.0
      └─ tmp@0.2.7 overridden
```

Observed validation:

- `npm audit`: exit 0, zero findings.
- `gts --version`: `7.0.0`.
- `external-editor` created, opened with a noninteractive test editor, and cleaned up a temporary file successfully.

Caveat: `external-editor@3.1.0` declares `tmp ^0.0.33`, so the override intentionally crosses its declared semver range. The temporary-file API used by `external-editor` worked in the smoke test, but the full KISS lint, typecheck, test, and VSIX packaging checks should gate adoption.

### 2. Remove or replace `gts` (preferred long-term)

`gts` is the only direct dependency in the chain, and its current release remains 7.0.0 with the legacy `inquirer` dependency. Removing it eliminates the chain rather than overriding an abandoned transitive constraint. This needs a separate tooling cleanup because KISS currently reads `gts/.prettierrc.json` and obtains some lint packages transitively through `gts`; those requirements would need explicit replacements.

### 3. Upstream resolution

An upstream release of `external-editor` that accepts a safe `tmp`, followed by compatible `inquirer` and `gts` releases, would avoid an override. At the snapshot date, no such release chain is available and ordinary `npm audit fix` reports no fix.

## Recommended actions

1. **Now:** adopt the targeted `external-editor > tmp@0.2.7` override in a focused dependency change and regenerate the lockfile.
1. **Before merging:** run `npm ci`, full `npm audit`, `npm audit --omit=dev`, extension typecheck/lint/tests, and VSIX packaging. Confirm the full audit is zero.
1. **CI/release:** add an explicit full audit gate (for example `npm audit --audit-level=high`) instead of relying on the post-install output currently disabled by `--no-audit`. Keep the production-only audit as a separate runtime-exposure signal.
1. **Documentation:** update the stale release-script comment so it no longer characterizes the chain as only old Low findings.
1. **Long-term:** replace `gts` with explicit maintained lint/format dependencies or upgrade when upstream removes the legacy interactive chain.
1. **Ongoing:** rerun the lockfile-based audit regularly because advisory metadata and available versions change independently of repository commits.
