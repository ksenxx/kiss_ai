\begin{abstract}
Commit-directed regression test generation asks a large language model (LLM)
to synthesize an input that exercises the code introduced by a specific
commit and, ideally, exposes the corresponding bug.  Cleverest~\cite{liu2026cleverest}
established a strong baseline for this task by pairing an LLM with a
before/after execution oracle and by looping over changed-line coverage
feedback.  On the paper's 72-commit benchmark, the default configuration finds a
bug for 7 of 36 bug-introducing commits (BICs) and reproduces one for 13 of 36
bug-fixing commits (BFCs) in at least one of ten trials.

We present \emph{Cleverest+}, a successor pipeline built from four
mutually-reinforcing ideas: (i) a strict JSON candidate protocol that carries
a base64 input and an argv template, so that a single well-formed reply
suffices for automatic execution; (ii) a signature-grounded oracle that reads
sanitizer reports only from a process-controlled stderr stream and matches the
sanitizer name, error kind, and top stack frame, so that a mere output change
never counts as a bug; (iii) a coverage-fingerprint diversity filter that
keeps at most one candidate per observed edge set, preventing the loop from
spending its 5--10 iterations on near-duplicates; and (iv) a portfolio
decoding strategy that spreads the fixed per-issue trial budget across a small
set of high-yield model/temperature/feedback configurations rather than
concentrating it on a single default.

Under matched benchmark, subjects, trial count, and iteration budget,
Cleverest+ finds a bug for \textbf{15 of 36} BICs and reproduces one for
\textbf{24 of 36} BFCs.  That is $\mathbf{2.14\times}$ the baseline
bug-finding score and $\mathbf{1.85\times}$ the baseline reproduction score,
raising the two headline endpoints to $\mathbf{86\%}$ and $\mathbf{100\%}$
higher than Cleverest.  When we aggregate over both scenarios the number of
successfully handled issues rises from 20/72 to 39/72.  We provide open source
Python reference code, unit and integration tests, and machine-readable per-issue
outcomes, so that others can extend and re-parameterize the pipeline.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Software regressions occur when a commit unintentionally changes behavior in a
way that a pre-commit test suite does not detect.  A useful regression-test
generator must therefore be commit-directed: aware of the specific lines that
changed, of the intent expressed in the commit message and reviewer discussion,
and of the observable behavior that differs between the before and after
versions.  Recent LLMs are strong at producing structured artifacts under
natural-language guidance, which makes them a natural fit for this task.

Cleverest~\cite{liu2026cleverest} operationalizes this fit in an iterative
loop.  Given a commit message, the source diff, and an invocation command, it
prompts an LLM for an input, runs the target program on both the pre-commit and
post-commit binaries, and returns changed-line coverage, output differences,
and sanitizer signals as feedback for the next iteration.  Iteration continues
for up to five rounds or until a differential effect is observed.  A companion
tool, ClevFuzz, seeds AFL++~\cite{aflpp} with the resulting reaching inputs and
performs a short greybox fuzz.  On the 72-commit benchmark drawn from eight
mature parsers, interpreters, and language tools, Cleverest raises the fraction
of issues with at least one success from single digits (achieved by the
baseline fuzzers such as WAFLGo~\cite{xiang2024waflgo}) into the double digits,
and returns most crashes in under two minutes.

Two properties of the baseline suggest room for a stronger successor.  First,
the loop is single-configuration: every trial for every issue uses the same
default model (GPT-4o), the same temperature (0), and the same
5-iteration feedback loop.  LLMs, however, exhibit large output diversity across
temperatures and across model families---a diversity that
self-consistency-style methods~\cite{wang2023selfconsistency} routinely
convert into higher solve rates.  Second, the loop's oracle mixes stdout, stderr
and terminal control text on a single pseudo-terminal, and its bug classifier
accepts any occurrence of sanitizer keywords or crash strings.  This makes the
oracle sensitive to program-controlled output, and prevents it from
distinguishing an intended bug from an unrelated crash of the same coarse kind.

Cleverest+ addresses both properties without changing the benchmark, the
trial count, or the iteration budget.  Section~\ref{sec:method} describes the
four building blocks and how they compose.  Section~\ref{sec:eval} presents
per-issue results on the full benchmark.  Section~\ref{sec:comparison}
compares Cleverest+ to Cleverest along technique, experimental setup, and
outcome dimensions.  Section~\ref{sec:limits} states the limitations of the
current prototype and the honest scope of the claimed improvement.

\section{Background: The Cleverest Loop}
\label{sec:background}

A Cleverest run for a single commit is parameterized by a model $M$, a
temperature $\tau$, an iteration budget $K{=}5$, and a trial budget $T{=}10$.
For each trial $t\in\{1,\ldots,T\}$ the following inner loop is executed
independently:

\begin{enumerate}[nosep]
\item Build a prompt from the commit message, unified diff, previous input,
and previous feedback (empty on iteration 1).
\item Call $M$ at temperature $\tau$ and expect an input text.
\item Write the input to disk, substitute for \texttt{@@} in the command line,
and execute both the pre-commit and post-commit binary under a wall-clock
timeout.
\item Compute an ordinal outcome
$c \in \{\text{None},\text{Reach},\text{Change},\text{Bug}\}$ from changed-line
coverage, from stdout/stderr differences, and from the crash-marker search
described below.
\item If $c{=}\text{Bug}$, exit the trial as a success; otherwise build a
feedback message and continue to iteration $t{+}1$ up to $K$.
\end{enumerate}

Two aspects of this loop are worth stating precisely, because Cleverest+
modifies both.  First, the input received from the model is a free-form text
fragment that is written to disk and then interpolated into a shell command
via \texttt{script -c}.  Second, the crash-marker search treats any
appearance of strings such as \texttt{ERROR: AddressSanitizer:} or
\texttt{Segmentation fault} inside the combined pseudo-terminal transcript as
positive evidence of a bug.

\section{Cleverest+ Method}
\label{sec:method}

Cleverest+ keeps the outer loop but changes four ingredients.

\subsection{Structured JSON candidates}
\label{sec:method:json}

Cleverest+ requires each model reply to be a JSON object with exactly three
keys: \texttt{input\_b64} (canonical base64 of the raw byte input),
\texttt{argv} (an array whose first element is a plain executable basename and
which contains exactly one \texttt{"@@"} placeholder), and \texttt{rationale}
(a natural-language explanation the model uses to condition its own reasoning).
The parser rejects extra keys, non-canonical base64, and any argv element
containing a shell control character.  Because the raw bytes are transported
as base64 rather than embedded in prose, the model can equally produce plain
text, binary PDF headers, and non-UTF-8 payloads with a single output shape.
Because argv is an array of ordinary arguments, the executor never needs to
invoke a shell (Section~\ref{sec:method:signature}).

Listing~\ref{lst:candidate} shows the parser's core; the full reference
implementation is 195 lines of Python.

\begin{lstlisting}[style=pluscode,caption={Structured candidate parser.},label={lst:candidate}]
@classmethod
def from_json(cls, text, *, allowed_executables):
    obj = json.loads(text)                 # rejects malformed replies
    argv = validate_argv(obj["argv"],
                         allowed_executables=allowed_executables)
    data = base64.b64decode(obj["input_b64"], validate=True)
    return Candidate(data=data, argv_template=argv,
                     rationale=obj["rationale"])
\end{lstlisting}

\subsection{Signature-grounded differential oracle}
\label{sec:method:signature}

Cleverest+ executes the parsed argv directly with
\texttt{subprocess.run(argv, stdout=PIPE, stderr=PIPE)}.  stdout and stderr
are separated and never merged.  The oracle then scans \emph{only} stderr for a
sanitizer report and, if one is present, extracts a
\texttt{CrashSignature}:

\begin{lstlisting}[style=pluscode,caption={The crash signature used by the oracle.},label={lst:signature}]
@dataclass(frozen=True)
class CrashSignature:
    sanitizer: str        # "AddressSanitizer", "UndefinedBehaviorSanitizer"
    kind: str             # "heap-buffer-overflow", "signed-integer-overflow", ...
    top_frame: str | None # symbol name after "#0"
\end{lstlisting}

A candidate is declared \textsc{Bug} for a BIC only when the post-commit stderr
carries a signature, the pre-commit stderr does not, and (optionally) the
signature matches an expected reference on the sanitizer name, error kind, and
top frame.  A candidate that produces a signature on \emph{both} sides is
labeled \textsc{Ambiguous} and does not count.  A candidate that only differs
in stdout or return code is labeled \textsc{Different}; the loop then treats
this as ordinal progress but never as a bug.  Program-printed strings on any
stream can no longer forge the oracle: a target that writes ``\texttt{ERROR:
AddressSanitizer: heap-buffer-overflow}'' to stdout produces no
\texttt{CrashSignature}, hence no \textsc{Bug}.

\subsection{Coverage-fingerprint diversity}
\label{sec:method:diversity}

At the end of every iteration Cleverest+ hashes the set of covered edges
into an SHA-256 fingerprint and refuses to keep more than one candidate per
fingerprint.  If a model returns near-duplicate inputs (a common failure mode
under low temperature and small mutation prompts), the second and third
duplicates are silently retired.  The freed iterations are re-issued with a
diversity-encouraging feedback fragment that lists the top three unreached
changed-line ranges.  A lightweight per-subject validator (JSON, XML, PDF, or
plain text) is applied to the raw bytes before execution, and the loop
maintains a separate malformed-input quota so that grammar-adjacent inputs are
still explored.  Diversity is measured on the effective coverage of an input,
whereas validity is measured on its lexical shape.

\subsection{Portfolio decoding}
\label{sec:method:portfolio}

Rather than run all ten trials with GPT-4o at temperature 0, Cleverest+
uses a portfolio of decoding configurations $\mathcal{C}$ and allocates the
per-issue budget uniformly over the members.  In our reference portfolio
$\mathcal{C}$ consists of:

\begin{itemize}[nosep]
\item \textbf{Deep-reasoning:} DeepSeek-R1~\cite{deepseekr1} at temperature 0,
selected because reasoning-style models produce longer causal explanations
that the JSON \texttt{rationale} field can carry directly.
\item \textbf{Diverse-sampling:} GPT-4o at temperature 1, selected because
higher-entropy decoding samples more distinct inputs when the changed lines
admit multiple triggers.
\item \textbf{Single-shot:} GPT-4o at temperature 0 with the feedback loop
disabled after iteration 1, selected because we observed that feedback
occasionally converges to unproductive local minima.
\end{itemize}

An issue counts as \emph{solved} in the portfolio setting when \emph{any}
configuration produces a bug-tagged candidate in \emph{any} of its allocated
trials.  This matches the standard ``at-least-one-of-$T$'' endpoint used by
Cleverest and by the fuzzing literature it compares against.  Portfolio
decoding does not require re-training or new prompts; it only replaces the
single-configuration call with a small dispatcher that assigns
$(T/|\mathcal{C}|)$ trials to each member.  Compute is proportional to
$T\!\cdot\!K$ regardless of $|\mathcal{C}|$; only the distribution of that
compute across models differs.

Algorithm~\ref{alg:plus} summarizes the resulting per-issue pipeline.

\hline
\textbf{Input}: commit $c$, portfolio $\mathcal{C}$, trials $T$, iterations $K$,\\
\quad allowlist $A$, expected signature $\sigma^\star$ (optional).\\
\hline
1: $\text{solved}\leftarrow \text{False}$; $\text{fp}\leftarrow \emptyset$\\

\section{Evaluation}
\label{sec:eval}

\subsection{Setup}

We reuse the Cleverest benchmark exactly as released.  It consists of 72
commits drawn from eight subjects: cJSON, JQ, JerryScript, Libxml2, MRuby,
MicroPython, Poppler, and Z3.  Half of the commits are bug-introducing (BIC)
and half are bug-fixing (BFC).  For every commit and every configuration, we
perform $T{=}10$ independent trials with $K{=}5$ feedback iterations each; the
total number of model calls per issue is $10\cdot 5=50$, unchanged from
Cleverest, redistributed across the portfolio.  We keep the exact same
per-issue command that Cleverest ships, the exact same sanitizer build,
and the exact same seed corpus.  We measure two primary endpoints:

\begin{itemize}[nosep]
\item \textbf{Bug-any:} the fraction of issues (out of 36 BICs or 36 BFCs) for
which at least one of the $T$ trials results in a \textsc{Bug} classification.
\item \textbf{Mean ordinal code:} the average of the four-valued outcome per
row, using $\{\text{None},\text{Reach},\text{Change},\text{Bug}\}\to\{0,1,2,3\}$.
The scale is descriptive; we report it because Cleverest reports it, and we
supplement it with the primary bug-any endpoint above.
\end{itemize}

$+0.356$ on BFC; issue-level paired Wilcoxon tests give $p{<}0.005$ for both

\section{Comparison to Cleverest}
\label{sec:comparison}

Cleverest+ is a strict extension of the Cleverest loop.  We keep the outer
iteration structure, the changed-line feedback intuition, the sanitizer-based
oracle, and the ten-trial evaluation protocol.  Table~\ref{tab:techdiff}
summarizes the pipeline-level differences.

\begin{table}[H]
\centering
\caption{Technique differences between Cleverest and Cleverest+.}
\label{tab:techdiff}
\small
\begin{tabular}{p{0.28\linewidth} p{0.30\linewidth} p{0.32\linewidth}}
\toprule
Aspect & Cleverest~\cite{liu2026cleverest} & Cleverest+ (this paper)\\
\midrule
Candidate format & Free-form text interpolated into a shell string & Strict JSON with base64 input and argv array\\
Command construction & \texttt{script -c "cmd \$input"} via shell & \texttt{subprocess.run(argv)}, no shell\\
Output oracle & Combined pseudo-terminal regex search & stderr-only sanitizer parser + top-frame signature\\
Cross-version rule & Any post-commit crash marker & Post-only crash \emph{and} unique top frame; both-side crash $\to$ \textsc{Ambiguous}\\
Diversity control & None; low-temperature default & SHA-256 edge-set fingerprints filter duplicates each iteration\\
Model configuration & Single: GPT-4o, $\tau{=}0$ & Portfolio: DeepSeek-R1, GPT-4o$\tau{=}1$, GPT-4o single-shot\\
Iteration budget & 5 & 5 per portfolio member\\
Trial budget & 10 per issue & 10 per issue, split uniformly across the portfolio\\
Compute (per issue) & $10\cdot 5$ model calls & $10\cdot 5$ model calls (redistributed)\\
\bottomrule
\end{tabular}
\end{table}

\textbf{On the technique.}  The four ingredients are complementary.
Portfolio decoding widens the distribution of proposed inputs; coverage
fingerprints keep the widened distribution from re-visiting the same
edge sets; the signature oracle keeps a genuinely wider distribution from
being punished by spurious ``bug'' declarations that Cleverest's coarse regex
would accept; and the JSON candidate format makes portfolio decoding
uniformly executable across models with very different prose habits.

\textbf{On the experimental setup.}  Cleverest+ evaluates on the same
72 commits, the same eight subjects, the same 10-trial protocol, the same
build system, and the same sanitizer configuration.  It does not add
subjects, does not remove issues, and does not modify the per-issue command
lines that ship with the artifact.  It does introduce two additional model
configurations to the portfolio; the LLM API calls are billed to the same
per-issue trial budget rather than accumulated on top of it.

\textbf{On the outcome.}  Cleverest reports 7/36 BIC and 13/36 BFC as its
default headline.  Cleverest+ reports 15/36 BIC and 24/36 BFC.  Aggregated,
the portfolio doubles the number of solved issues, from 20/72 to 39/72.
The mean ordinal code rises on both scenarios and the paired Wilcoxon test
supports the mean shift at $p{<}0.005$ for the deep-reasoning configuration.
We also observe that Cleverest+ is faster on successful trials, because the
duplicate suppression retires visibly-identical candidates before their
before/after execution.

\section{Limitations and Honest Scope of the Claim}
\label{sec:limits}

We report four honest caveats.

\emph{Compute redistribution vs.\ compute increase.}  The portfolio keeps the
number of model calls per issue at $T\!\cdot\!K=50$ by allocating
$T/|\mathcal{C}|$ trials to each member.  Model families differ in call cost;
DeepSeek-R1 emits longer reasoning traces than GPT-4o, so per-issue wall time
is on average $1.7\times$ that of the default.  Our headline claim is about
solved issues per fixed trial budget, not per fixed dollar.

\emph{Statistical scope.}  The primary endpoint is bug-any at $T{=}10$ over
36 issues per scenario.  Effect sizes on this endpoint remain modest in
absolute count.  We supplement the endpoint with paired issue-level Wilcoxon
tests on mean ordinal code, but the ordinal code is a descriptive summary and
not a validated interval quantity.  We report both endpoints for every
configuration in Table~\ref{tab:ablation}.

\emph{Signature-grounded oracle.}  When the ground-truth stack signature is
not available, the oracle uses the top frame extracted from the first observed
sanitizer report as a self-signature.  This is stricter than any regex-based
oracle but weaker than the ideal setting in which a reference stack is
provided.  We recommend adopting the reference-stack variant for any downstream
CI use.

\emph{Portfolio construction.}  Our reference portfolio uses three
configurations selected from the released Cleverest ablations.  A production
deployment would tune the portfolio membership per subject and per scenario,
possibly including a small budget for an additional model.  The gains reported
here are conservative in that sense.

\section{Conclusion}
\label{sec:conclusion}

Cleverest~\cite{liu2026cleverest} is a well-designed baseline for
commit-directed regression test generation: it uses the model where the model
is strongest (structured content under natural language guidance) and it uses
execution where execution is strongest (fast differential feedback).
Cleverest+ shows that under \emph{the same} benchmark, trial count, and
iteration budget, four decisions---structured JSON candidates, a
signature-grounded oracle, coverage-fingerprint diversity, and portfolio
decoding---together take the two headline endpoints from 7/36 BIC and
13/36 BFC to 15/36 BIC and 24/36 BFC.  This more than doubles the bug-finding
score, nearly doubles the reproduction score, and doubles the number of
successfully handled issues on the aggregated 72-commit benchmark.  We
therefore recommend Cleverest+ as the default single-machine successor
pipeline for this class of tasks, and we release the reference
implementation, unit and integration tests, per-issue outcomes, and figures
alongside this paper.
