## Related Work

### Distributed and Large-Scale de novo Genome Assembly

Early work on memory-efficient de Bruijn graph construction explored data structures that shrink the k‑mer representation to make de novo assembly feasible on commodity servers. SparseAssembler introduced a sparse de Bruijn graph that skips intermediate k‑mers to reduce memory from O((2k+8)N) to approximately O((2k/g+8)N), also incorporating graph-based denoising to tolerate sequencing errors [1](https://arxiv.org/abs/1106.2603). While this significantly cuts memory, it still assumes conventional CPU-centric execution and does not fundamentally address memory bandwidth limits or irregular data access at scale.

To handle population-scale datasets, several assemblers were redesigned for massive distributed-memory machines. HipMer parallelizes the Meraculous pipeline end‑to‑end for extreme-scale whole-genome assembly, carefully engineering distributed hash tables and communication patterns to assemble human and large plant genomes on tens of thousands of cores [2](https://arxiv.org/abs/1705.11147). MetaHipMer extends this approach to de novo metagenome assembly, combining an iterative de Bruijn graph strategy with specialized scaffolding for complex microbiomes, and demonstrating strong scaling on multi-terabyte environmental datasets [3](https://arxiv.org/abs/1809.07014). These systems highlight that the dominant costs arise from memory-bound k‑mer counting, graph construction, and traversal over massive, irregular data structures. However, they remain built on traditional CPU clusters, so their performance and memory footprint are fundamentally constrained by off-chip DRAM bandwidth and capacity.

### Near-Data and In-Storage Acceleration for Genome Analysis

Recognizing that data movement dominates energy and time costs, recent work has explored pushing computation closer to where genomic data resides. GenStore proposes an in-storage computing system that embeds lightweight accelerators and filters inside NAND flash SSDs to offload large portions of the read-mapping pipeline [4](https://arxiv.org/abs/2202.10400). By co-designing SSD-resident filters and data layouts with the host software, GenStore reduces both data transfer and compute overheads, achieving multi-fold speedups and energy savings. While architecturally distant from near-memory DRAM, GenStore exemplifies how hardware/software co-design around storage hierarchies can shrink memory traffic for genome analysis.

Complementary to in-storage computing, several works leverage processing-in-memory (PIM) or near-memory compute for seed filtering and alignment. GRIM‑Filter introduces a seed-location filter tailored to 3D‑stacked memories with logic layers, using massively parallel in-memory bitwise operations to eliminate most candidate locations before alignment [5](https://arxiv.org/abs/1711.01177). RAPIDx targets dynamic-programming-based sequence alignment with a ReRAM PIM architecture and an adaptive banded algorithm co-designed to fit the PIM substrate, yielding large throughput and energy gains over CPU, GPU, and earlier PIM designs [6](https://arxiv.org/abs/2211.05733). DART‑PIM presents a digital PIM framework for end-to-end DNA read mapping, reorganizing indexing, filtering, and alignment around the constraints and opportunities of PIM arrays [7](https://arxiv.org/abs/2411.03832). These accelerators demonstrate the value of exploiting high internal memory bandwidth, but focus on read mapping or alignment kernels rather than full de novo assembly pipelines, and they typically operate over relatively regular dynamic-programming or bitvector computations.

### Toward Hardware/Software Co-design for Assembly

The above PIM and in-storage systems primarily target read mapping stages where access patterns, although large, are more structured. In contrast, de novo assembly requires managing massive, sparse, and dynamically evolving graph structures, which amplify issues of load imbalance, pointer chasing, and memory capacity pressure. Existing distributed assemblers such as HipMer and MetaHipMer already expose considerable parallelism but ultimately rely on general-purpose CPUs and host memory [2](https://arxiv.org/abs/1705.11147)[3](https://arxiv.org/abs/1809.07014). At the same time, low‑level de Bruijn data-structure innovations like SparseAssembler optimize memory usage but do not exploit specialized hardware capabilities [1](https://arxiv.org/abs/1106.2603).

NMP‑PaK differs from these prior efforts by explicitly co-designing a channel-level near-memory processing architecture and the de novo assembly software pipeline. Instead of accelerating only seed filters or alignment subroutines, it targets the scalability bottlenecks of graph-centric assembly itself: the construction and manipulation of large, interdependent data structures under highly irregular access patterns. Unlike GenStore’s flash-centric design [4](https://arxiv.org/abs/2202.10400) or PIM accelerators specialized for alignment [5](https://arxiv.org/abs/1711.01177)[6](https://arxiv.org/abs/2211.05733)[7](https://arxiv.org/abs/2411.03832), NMP‑PaK restructures de novo assembly around channel-local scratchpads and custom processing elements, and further introduces batch scheduling and hybrid CPU–NMP execution to mitigate hardware underutilization. In doing so, it bridges the gap between distributed CPU-only assemblers and fine-grained PIM accelerators, providing a holistic near-memory solution for end-to-end, large-scale de novo genome assembly.
