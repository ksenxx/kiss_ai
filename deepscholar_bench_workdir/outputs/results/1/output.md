# Related Work

## LLM Serving Systems and Memory Management
Efficient serving of large language models has become a central systems challenge as cloud providers move billions of requests per day through transformer stacks. vLLM's PagedAttention introduced virtual-memory-style paging of the key–value (KV) cache to eliminate fragmentation and enable dynamic batching, and has become the de-facto backbone of many production stacks [1](https://arxiv.org/abs/2309.06180). vAttention revisited this design and showed that contiguous virtual memory with demand-paged physical backing can match paging benefits while retaining kernel simplicity [2](https://arxiv.org/abs/2405.04437). FastServe attacks head-of-line blocking via token-level preemptive scheduling and a skip-join MLFQ, coupled with GPU–host state migration [3](https://arxiv.org/abs/2305.05920). Sarathi-Serve tames the throughput–latency trade-off through chunked prefills and stall-free scheduling [4](https://arxiv.org/abs/2403.02310). Splitwise [5](https://arxiv.org/abs/2311.18677) and DistServe [6](https://arxiv.org/abs/2401.09670) argue for disaggregating the compute-bound prefill and memory-bound decode phases across pools of accelerators, each of which imposes distinct pressure on KV cache placement and transfer. FlexGen further explores high-throughput single-GPU generation via aggressive offloading of weights and KV state to CPU and disk [7](https://arxiv.org/abs/2303.06865).

## KV Cache Reuse and Prefix Sharing
A large body of work exploits the observation that many requests share prefixes—system prompts, few-shot exemplars, retrieved documents, or prior conversational turns. SGLang's RadixAttention organizes cached KV tensors in a radix tree so that arbitrary common prefixes can be discovered and reused across concurrent programs [8](https://arxiv.org/abs/2312.07104). Prompt Cache formalizes reusable prompt "modules" via a schema and precomputes their attention states for cross-request reuse [9](https://arxiv.org/abs/2311.04934). ChunkAttention breaks monolithic KV tensors into chunks indexed by a prefix tree and co-designs a two-phase attention kernel for prefix sharing [10](https://arxiv.org/abs/2402.15220), while Hydragen reformulates attention over shared prefixes as batched matmuls to accelerate large-batch decoding [11](https://arxiv.org/abs/2402.05099). Preble scales prefix reuse to distributed deployments by co-optimizing KV-state locality with load balancing [12](https://arxiv.org/abs/2407.00023). CachedAttention persists per-session KV state across multi-turn conversations, avoiding recomputation of dialogue history [13](https://arxiv.org/abs/2403.19708), and EFIM refines this idea for infilling workloads where reuse boundaries are non-trivial [14](https://arxiv.org/abs/2505.21889). Our study complements these systems by showing, on a real production trace, that single-turn cross-request reuses are as important as multi-turn reuses—an assumption not covered by prior synthetic evaluations.

## Hierarchical and Disaggregated KV Storage
Because on-GPU HBM is scarce, several systems build hierarchical KV pools spanning HBM, host DRAM, SSD, and network. MemServe manages an elastic KV memory pool tailored to prefill/decode disaggregation [15](https://arxiv.org/abs/2406.17565); Mell extends this to multi-GPU memory-efficient reuse [16](https://arxiv.org/abs/2501.06709). CacheGen encodes KV tensors for compact storage and fast network streaming so that cached context can be shipped between nodes faster than recomputing it [17](https://arxiv.org/abs/2310.07240), and DéjàVu streams KV state for fault-tolerant, pipeline-parallel serving [18](https://arxiv.org/abs/2403.01876). Our findings on the moderate cache size needed for near-ideal hit ratios directly inform the sizing and admission policies for such tiered stores.

## KV Cache Compression and Eviction
Orthogonal to reuse, another line of work reduces the per-request KV footprint. H2O identifies "heavy-hitter" tokens whose keys dominate attention and evicts the rest [19](https://arxiv.org/abs/2306.14048); SnapKV compresses long-context caches by anticipating which tokens future queries will attend to [20](https://arxiv.org/abs/2404.14469); StreamingLLM shows that retaining a few initial "attention-sink" tokens plus a sliding window preserves quality over unbounded streams [21](https://arxiv.org/abs/2309.17453). InfiniGen dynamically pages the KV cache with speculative attention estimation [22](https://arxiv.org/abs/2406.19707), while KIVI [23](https://arxiv.org/abs/2402.02750) and Oaken [24](https://arxiv.org/abs/2503.18599) quantize KV tensors to 2–4 bits with online/offline hybrids. These intra-request policies are complementary to the inter-request, workload-aware eviction policy we propose.

## Workload Characterization
Realistic traces are essential for evaluating the systems above. BurstGPT released ten million Azure OpenAI traces and highlighted concurrency and conversation patterns [25](https://arxiv.org/abs/2401.17644), but its focus is request arrivals and lengths rather than KV reuse structure. Our work provides, to our knowledge, the first systematic characterization of KV-cache workload patterns—reuse skew, reuse-time distributions, and category-level predictability—from a leading commercial LLM service, and translates these observations into a workload-aware eviction policy.
